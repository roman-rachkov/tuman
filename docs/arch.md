# АРХИТЕКТУРНЫЙ ДОКУМЕНТ
## «Ржавый туман» – политическая песочница с локальной LLM

**Версия 1.0** (соответствует GDD v0.1 → v1.0)

---

## 1. Общие принципы архитектуры

1. **Модульность** – каждый крупный блок (граф, агенты, заявки, LLM, UI) изолирован, имеет чёткий интерфейс и может разрабатываться параллельно.
2. **Тестируемость** – бизнес-логика вынесена из UI, использует чистые функции и моки для внешних зависимостей.
3. **Производительность** – тяжёлые вычисления (A*, LLM инференс) вынесены в Web Workers.
4. **Состояние как единственный источник истины** – глобальное состояние игры хранится в централизованном хранилище (Redux / MobX) с иммутабельными обновлениями.
5. **Событийно-ориентированная коммуникация** – компоненты общаются через события / команды, уменьшая связность.
6. **Инкрементальная сложность** – архитектура позволяет на MVP использовать упрощённые реализации (моки, синхронные вызовы) и подменять их на полноценные в более поздних версиях.

## 3. Ключевые абстракции и интерфейсы

### 3.1. Граф (core/graph)

```ts
interface Node {
  id: string;
  lat: number;
  lng: number;
  tags?: Record<string, string>;
}

interface Edge {
  id: string;
  from: string;
  to: string;
  length: number;        // метры
  baseSpeed: number;     // м/с
  state: 0 | 1 | 2 | 3;  // intact/damaged/blocked/requires_protection
}

class Graph {
  loadFromJSON(data: any): void;
  getShortestPath(start: string, end: string, filter?: (edge: Edge) => boolean): string[];
  updateEdgeState(edgeId: string, newState: number): void;
}
```

**A* реализован как чистая функция в `pathfinder.ts`, принимающая граф, хэш-функцию и эвристику.**

### 3.2. Агент (core/agents)

```ts
interface Agent {
  id: string;
  name: string;
  currentNodeId: string;
  state: 'idle' | 'moving_to_start' | 'executing' | 'moving_to_end';
  inventory: Map<ResourceType, number>;
  skills: { repair: number; combat: number; stealth: number };
  needs: { hunger: number; thirst: number };
  personality: 'greedy' | 'altruist' | 'coward';
  currentJobId?: string;
  targetNodePath?: string[];
  positionLatLng?: { lat: number; lng: number }; // интерполированное положение для анимации
}
```

**AgentBrain** – чистая функция, принимающая агента и список активных заявок, возвращающая `jobId` или `null`.

### 3.3. Заявка (core/jobs)

```ts
type JobType = 'delivery' | 'repair' | 'elimination' | 'scouting' | 'guard' | 'sabotage';

interface Job {
  id: string;
  type: JobType;
  status: 'pending' | 'assigned' | 'completed' | 'failed';
  reward: Map<ResourceType, number>;
  targetLocation: string;   // id здания или ребра
  targetResource?: ResourceType;   // для доставки: что везти
  sourceLocation?: string;         // откуда взять
  expiresAt?: number;              // игровое время
  requiredSkills?: Partial<AgentSkills>;
}
```

**JobManager** отвечает за создание, отмену, резервирование награды.

### 3.4. Состояние игры (Redux store)

Структура состояния (пример slice):

```ts
interface GameState {
  version: string;
  currentTime: number;           // игровые часы с начала партии
  world: {
    districts: Record<string, DistrictState>; // каждый район
  };
  agents: Record<string, Agent>;
  jobs: Record<string, Job>;
  resources: Record<string, ResourcePool>;    // по зданиям
  factions: Record<string, FactionReputation>;
  ui: {
    selectedAgentId?: string;
    selectedBuildingId?: string;
    modal: null | { type: 'newJob' | 'event' ; data: any };
  };
}
```

Все изменения производятся через **dispatch(action)**. Асинхронные действия (движение, LLM) используют **thunks** или **sagas**.

## 4. Потоки данных и взаимодействие компонентов

### 4.1. Игровой цикл (сервисы)

Главный цикл запускается в `GameService.start()`. Каждый **тик** (например, раз в 100 мс реального времени) обновляет:

- **MovementService** – пересчитывает позиции всех агентов, которые находятся в состоянии `moving_*`. Если агент достиг узла – генерирует событие `ARRIVED_AT_NODE`.
- **EconomicService** – раз в игровой час списывает ресурсы на потребление населения, обновляет коррозию рёбер (вероятностно).
- **JobAssignmentService** – для каждого свободного агента вызывает `AgentBrain` и назначает заявку (через `JobManager.assignJob`).
- **EventDispatchService** – проверяет триггеры для генерации LLM-событий (таймер, достижение порога ресурсов). Генерирует событие и ставит в очередь модальное окно.

**Коммуникация через события:** сервисы не вызывают UI напрямую, а диспатчат действия в store. UI подписывается на изменения через `useSelector`.

### 4.2. Интеграция LLM

- Пользователь нажимает «Предзагрузить LLM» → `LLMClient.preload()` загружает модель в фоновом воркере, обновляет статус в store.
- При наступлении триггера `EventDispatchService` вызывает `EventGenerator.generateEvent(context)`, который:
    - Если LLM готова – отправляет промпт в `llm.worker.ts`, получает JSON, валидирует через XGrammar.
    - Если нет – берёт случайное событие из `FallbackEvents`.
- Полученное событие (с choices) записывается в store как `pendingEvent`. UI показывает модальное окно.
- После выбора игрока `EventDispatchService` применяет эффекты (изменяет ресурсы, репутацию, состояние рёбер) через store dispatch.

### 4.3. Анимация движения

- `MovementService` работает отдельно от игрового цикла (на `requestAnimationFrame`), чтобы анимация была плавной.
- Для каждого движущегося агента хранится текущий сегмент (ребро), прогресс (0..1). Каждый кадр увеличивается `progress += speed * deltaTime / edge.length`.
- При `progress >= 1` переключается на следующий узел. Если путь закончен – генерируется событие `AGENT_COMPLETED_JOB`.
- Координаты для отображения вычисляются линейной интерполяцией между узлами.

## 5. Модульная подстановка для разных версий

Архитектура позволяет легко включать/выключать сложные фичи через **стратегии** или **условную компиляцию** (feature flags). Например:

- **Graph**: в v0.1 используется статический JSON, в v0.4 добавляется динамическое обновление `edge.state`. Класс `Graph` содержит методы `updateEdgeState`, которые в v0.1 просто ничего не делают.
- **AgentBrain**: в v0.1 – простая функция `return jobs[0]`. В v0.3 заменяется на полноценную формулу полезности.
- **LLMClient**: в v0.5 до этого просто заглушка, возвращающая `null`. MVP не требует LLM, всё через fallback.

Файлы конфигурации (`config.ts`) могут содержать флаги:

```ts
export const Features = {
  dynamicEdges: false,   // включить в v0.4
  population: false,     // v0.3
  llm: false,            // v0.5
  factionsAI: false,     // v0.7
};
```

При разработке следующей версии достаточно переключить флаг и реализовать недостающие модули.

## 7. Обработка ошибок и fallback'и

- **LLM не загрузилась** → используется fallback-события, уведомление в логе.
- **Агент не может построить путь** → заявка отменяется, агент переходит в idle, лог: «Нет пути до цели».
- **Ресурс на складе закончился во время выполнения заявки** → заявка проваливается, награда не выдаётся.
- **Сохранение повреждено** → при старте проверяется валидность; если нет – начать новую игру.

## 8. Производительность и оптимизации

- **Мемоизация селекторов** (reselect) для частых вычислений (список видимых агентов, доступные заявки).
- **Виртуализация** лога (только последние 100 сообщений).
- **Ограничение частоты обновления** для экономического сервиса (раз в секунду реального времени, а не каждый кадр).
- **A* кэширование** – результаты поиска между теми же узлами при неизменном графе.
- **LLM worker** запускается только при необходимости; модель выгружается из памяти, если не использовалась долгое время (опционально).

## 11. Заключение

Архитектура построена вокруг **модульного ядра**, независимого от UI, и **сервисного слоя**, управляющего игровым циклом. Такой подход позволяет команде разрабатывать параллельно разные системы (граф, агенты, LLM), легко тестировать и заменять реализации по мере роста сложности. Проект готов к итеративной разработке от простого MVP до полноценной политической песочницы.

**Документ обязателен к ознакомлению всеми участниками разработки.**
